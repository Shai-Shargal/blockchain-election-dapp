# Known Bugs and Limitations

## Privacy
- **Questionnaire votes are NOT private.** The candidateId appears in transaction calldata on-chain. Only the UI hides the result before submission. For true privacy, a ZK proof would be needed (out of scope).

## Merkle Tree
- The voter list is fetched from IPFS at voting time. If IPFS pinning lapses, proofs cannot be regenerated by new users (existing downloaded lists still work).

## Admin

- **Election times CAN be changed by the admin even after voting starts.** The contract does not prevent this. An admin could shorten the `endTime` to stop voting early, or extend it. This is a known limitation for the course project scope. A production contract would add `if (block.timestamp >= startTime && hasVotes) revert ElectionOngoing()` guard.

- **Admin can add candidates while voting is live.** `addCandidate` has no check against in-progress elections. Candidates added during voting are eligible to receive votes immediately. For this course project, the admin is trusted.

## Production IPFS Upload

The IPFS voter registry upload uses a Vite development-server plugin (`configureServer` in `vite.config.ts`). This plugin is **not included in production builds**. In production, calling `/api/upload` would return 404. For a live deployment, a minimal Express.js endpoint would need to wrap the Pinata upload call. For the course demo on Sepolia testnet with the Vite dev server, this works correctly.

## ERC20 Reward
- Reward amount is hardcoded at compile time (10 BAL). Changing it requires redeployment.
